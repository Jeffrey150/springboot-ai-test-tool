# Test AI 项目开发规范
- **test-ai-backend**：后端项目
- **test-ai-frontend**：前端项目

> 本规范适用于 Test AI 项目的所有开发人员，无论使用何种开发工具或 IDE。
> 技术栈：前端 Vue 3 + TypeScript + ElementPlus，后端 Java Spring Boot 2.7 + MyBatisPlus。

---

## 核心原则（必须严格遵守 ）

### 1. 不要假设，不要隐藏困惑
- **遇到模糊需求时，必须立即暂停并向用户确认**，不得擅自猜测或补充未提及的逻辑。
- **对不确定的API、字段、业务规则，必须提出澄清问题**后再继续编码。
- **禁止**基于"常见做法"或"通常如此"的假设编写代码。

### 2. 用最小代码解决问题
- **优先使用最简单、最直接的方式实现功能**，避免过度设计。
- **禁止**引入不必要的抽象层、设计模式或第三方库。
- 每一行代码都必须有明确的目的，删除任何冗余代码。

### 3. 只改必须改的
- **修改范围必须精确限定在需求范围内**，不得连带修改无关文件或逻辑。
- 重构仅在明确需要时进行，且必须单独说明。
- 保持现有代码风格和行为的一致性，不因个人偏好而改变。

### 4. 定义成功标准，循环验证
- **编写代码前，先明确定义该功能的验收标准**（输入、输出、边界条件、异常场景）。
- **每完成一个功能单元，立即进行自测验证**，确保符合成功标准。
- **验证不通过时，不得进入下一阶段**，必须修复后再继续。

## 技术栈规范

### 前端：Vue 3 + TypeScript
- **必须使用 Composition API + `<script setup>` 语法**。
- **所有变量、函数、组件 props 必须显式声明类型**，禁止使用 `any`。
- 组件命名使用 PascalCase，文件命名与组件名一致。
- 状态管理优先使用 Pinia，避免滥用全局状态。
- API 调用必须封装到 services/api 层，统一处理错误和 loading 状态。

### 后端：Java
- **使用 Spring Boot 框架**，遵循 RESTful API 设计规范。
- **所有接口必须校验入参**，使用 `@Valid` / `@Validated` 进行参数校验。
- **数据库操作必须使用参数化查询**，严禁字符串拼接 SQL（防 SQL 注入）。
- 异常必须统一处理，返回标准格式的错误响应（包含错误码和错误信息）。
- 日志记录必须包含上下文信息（如请求ID、用户ID），便于问题排查。

## 安全规范

### 输入安全
- **所有外部输入（用户输入、API 参数、文件上传）必须校验**：类型、长度、格式、范围。
- **输出到前端的数据必须转义**，防止 XSS 攻击。
- 文件上传必须校验文件类型、大小，禁止执行上传文件。

### 认证与授权
- **敏感接口必须校验用户身份和权限**。
- 密码必须使用强哈希算法（如 BCrypt）存储，禁止明文存储。
- Token 必须设置合理的过期时间，支持刷新机制。

### 数据安全
- **敏感数据（密码、密钥、Token）禁止硬编码在代码中**，使用环境变量或配置中心管理。
- 数据库连接字符串、API 密钥等禁止提交到版本控制。
- 生产环境必须启用 HTTPS，禁止明文传输敏感数据。

## 代码规划流程

每次编写代码前，按以下步骤执行：

1. **需求分析**：明确功能目标、输入输出、边界条件、异常场景。
2. **方案设计**：列出实现步骤，评估是否需要新建文件或修改现有文件。
3. **代码编写**：按最小代码原则实现，严格遵守技术栈规范。
4. **自测验证**：对照成功标准进行验证，确保功能正确、边界覆盖、异常处理完善。
5. **代码审查**：检查是否符合本规范的所有要求，特别是安全规范。

## 自测验证规范

### 验证清单（每次提交代码前必须完成）

- [ ] **功能验证**：核心功能是否按预期工作？
- [ ] **边界验证**：空值、极限值、超长输入等边界场景是否处理？
- [ ] **异常验证**：错误输入、网络异常、权限不足等异常场景是否有合理处理？
- [ ] **安全验证**：是否存在 SQL 注入、XSS、越权访问等安全风险？
- [ ] **类型验证**：TypeScript/Java 类型是否正确，编译是否通过？
- [ ] **代码规范**：是否符合项目命名规范、代码风格？

### 验证方式
- **前端**：在浏览器中运行，验证 UI 渲染、交互逻辑、API 调用。
- **后端**：编写单元测试或手动调用接口（Postman/curl），验证返回结果和异常处理。
- **全链路**：验证从前端到后端的完整流程是否通畅。

## 沟通规范

- **遇到任何不确定的问题，优先向用户确认**，不要自行决定。
- **修改现有代码时，必须说明修改理由和影响范围**。
- **引入新技术或依赖时，必须说明必要性和替代方案**。
- **代码中复杂逻辑必须添加注释**，说明"为什么"而非"做什么"。
